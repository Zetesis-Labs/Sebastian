# CROSS-CHECK

## Contradictions
- fw-gate-preroll-protocol vs agent-modernization: Outgoing RPC from C SDK 0.3.10 — gate-preroll verified against core/rpc_manager.h/c that `livekit_room_rpc_invoke` is NOT implemented (it only appears in a comment in livekit.h:210-212), while agent-modernization claims the SDK "already exposes `rpc_invoke`" citing only the header (livekit.h:423-454). The device→agent channel must be designed over `publish_data` with its own retries (gate-preroll version, the one verified in implementation), not over RPC with ack.
- fw-wakeword-spike vs fw-gate-preroll-protocol: Incompatible sample rate and pre-roll size — the spike proposes feeding the detector directly at 48 kHz without decimating (verified ESPHome pattern) and a pre-roll ring of 1.5 s = 144 KB, while gate-preroll assumes decimation ×3 to 16 kHz (hop of 160 samples) and a pre-roll of 2 s = 64 KB / 1.5 s = 48 KB (s16 @16k). This affects the ring size in PSRAM, the byte stream volume, and the consumers of ddsd-speaker-id (ECAPA and Speechmatics expect 16 kHz). The ROADMAP (line 71) assumes 16 kHz; the spike refutes this for the detector, but no one has decided the canonical pre-roll format.
- agent-modernization vs ddsd-speaker-id: Two different mechanisms for the same LINGER DDSD gate — agent-modernization designs `turn_detection=None` + `inference.TurnDetector` + `StopResponse` on `on_user_turn_completed`; ddsd-speaker-id designs `turn_detection="manual"` + `commit_user_turn()`/`clear_user_turn()`. They are mutually exclusive in the same session and have different implications (`StopResponse` doesn't work with server VAD; manual disables all automatic detection). One must be chosen before writing the gate.
- agent-modernization vs model-cost-matrix vs ddsd-speaker-id: Unreconciled primary model — agent-modernization builds on OpenAI Realtime (default `gpt-realtime` + `semantic_vad`), model-cost-matrix recommends Gemini/Nova for cost (~4x cheaper) and vetoes `gpt-realtime-2` due to plugin bugs (#5768, #5808), and the ddsd Proactive Audio shortcut requires `gemini-2.5-flash-native-audio` as the main model (it doesn't exist in 3.1 or in OpenAI). The chosen model determines what turn/gate/interruption mechanisms are available; it's not just a cost variable.
- infra-selfhost vs aec-verification: infra mitigates the loss of BVC in self-host with 'the post-AEC ASR beam from the XVF is already the clean signal', but aec-verification demonstrates that this beam (mux 7,3) does NOT have residual echo suppression — this is precisely the cause of the volume being capped at 35. Removing BVC before closing AEC degrades doubly; the dtln A/B testing must be done with the AEC finding already closed. Furthermore, `agent.py` uses unconditional `noise_cancellation.BVC()`, which fails outright against a self-hosted server.
- agent-modernization vs infra-selfhost: `inference.TurnDetector` v1 (audio-native, Spanish) runs only in LiveKit Inference (Cloud); the self-host plan loses it and only the local v1-mini on CPU remains. The P1 design of the agent-modernization gate depends on v1 — if the turn detector is not parameterized via config from day 1, the migration to the edge will break the gate.
- ddsd-speaker-id vs agent-modernization vs model-cost-matrix: Unreconciled parallel STT budget — ddsd assumes the Speechmatics branch with diarization (and 13-15 days of effort), agent-modernization says 'add parallel STT only if measurements justify it', and model-cost-matrix shows the gate is not justified by cost with cheap models (<$4/month for LINGER without gate). The gate is sustained only by UX/privacy: its priority and recurring cost (Speechmatics RT 24/7 in LINGER windows) must be explicitly decided.
- agent-modernization vs infra-selfhost: Token server phasing — agent-modernization includes it in its immediate effort (1 d, a requirement for explicit dispatch that eliminates the cold-room workaround), the ROADMAP places it in P0, but infra-selfhost relegates it to P2 ('meanwhile, mint long tokens with lk CLI'). gate-preroll also makes it a precondition to advance IDLE. Phase misalignment in a component that blocks three areas.
- fw-gate-preroll-protocol vs ROADMAP/infra-selfhost: The 'Opus DTX ≈ 0 network cost' assumption in ARMED is false — DTX only exists at 8/12/16 kHz (published at 48 kHz) and `enable_dtx` is not exposed by the SDK. And Cloud economics are measured in participant minutes (86,400/month connected 24/7 vs 5,000 in the free tier), not in kbps: real savings require IDLE with room disconnection, which in turn requires the token server.
- fw-wakeword-spike vs wakeword-training: Different arena size — the spike budgets 26,080 B (`okay_nabu`) while the custom trainer's v2 manifest declares `tensor_arena_size=30000`. Minor, but the firmware component must size the arena by reading the manifest (with 1.5x/2x probing), not hardcoding the spike's value.

## Gaps
- There is no confirmed custom LiveKit Cloud project with an API key/secret (the current token is from a sandbox, iss API5JKjV7RkL9AE, exp ~August 2026). Without keys, there is no token server, no explicit dispatch, and no dispatch testing — this blocks agent-modernization, gate-preroll (IDLE), and infra. This is an open question in agent-modernization but no area owns it as a task.
- There is no single specification for the device↔agent protocol: names of topics/RPC methods, JSON schemas, versioning, agent identity verification (`agent_name` convention vs `kind==AGENT`), and what travels via `publish_data` vs the byte stream. gate-preroll and agent-modernization implement each end with their own assumptions; without a frozen contract, E2E integration will fail on details (e.g., pre-roll topic and attributes that the C SDK cannot send).
- The IDLE/ARMED/ATTENDING/ENGAGED/LINGER state machine is not specified by anyone: who is the source of truth (firmware or agent), how is the state reconciled after a WiFi reconnection/agent crash, concrete timeouts, and what happens to the gate if the agent disappears with the gate open. Four areas cite it as context; none design it.
- The re-verification of the wake word in the agent (stage 2 of the two-stage design) has no owner and no decided engine: openWakeWord over the PCM, the microWakeWord `.tflite` itself in Python, or STT+textual match. The 200-500 ms veto budget assumed by gate-preroll (audio leak on false positive) is not validated by anyone.
- There is no consolidated firmware memory budget: fw-wakeword-spike adds its part (arena+frontend, 10-25 KB internal) and gate-preroll adds its part (48-144 KB PSRAM pre-roll ring, 8 KB StreamBuffer, invocation task), but no one has measured the free internal heap in steady state (estimated 60-120 KB) nor added both features on the same device. The spike GO with `okay_nabu` does not cover the total.
- The refactoring of `mic_src`/`xvf_ui` (separating gate vs `hard_mute`, removing the forced `setMuted` every 80 ms) is planned by TWO areas separately (fw-wakeword-spike and fw-gate-preroll-protocol) without a common design for the semantics (LED on = gate open; physical button = cuts the tap too). This must be done only once, before both, or each area will step on the other. Additionally, it contradicts ROADMAP lines 55-57 ('reuse `setMuted()` as gate'), which must be corrected.
- Where the agent and token server run in P0/P1 (dev Mac vs deployment at the edge) and how `/token` is exposed to the ESP32 (plaintext LAN vs TLS) — this conditions the Zig token fetch code at boot and no area decides it; infra-selfhost only covers the P2 scenario.
- CI and registry for the `sebastian-agent`/`sebastian-token` images (ghcr.io vs Harbor of px-socrates) and the build pipeline for the Sebastian repo: infra leaves it as an open question, no other area covers it. Blocks deployment at the edge.
- Privacy and audio retention policy not defined: permanent ddsd shadow mode (logs decisions and audio), voice enrollment WAVs, current unconditional WAV recording by `agent.py`, and the pre-roll sent to the cloud on false positives. No one defines retention, where it is saved, or how it is deleted.
- Decision to advance IDLE (room timeout + wake reconnection) to P0: raised as an open question by gate-preroll, agent-modernization, and infra-selfhost, and whether the Cloud free tier (5,000 min/month) is viable depends on it, but no one owns it or has measured the real reconnection latency (~1-2 s estimated).

## Integration Order

PHASE A — cheap parallel validators (week 1): (1) aec-verification Test 0 (far-end passthrough mux, 10 min) + read `AECCONVERGED`/`OP_R`: decides if full-duplex is possible and unblocks the barge-in plan; if it fails, the whole plan remains half-duplex and ddsd/LINGER change shape. (2) fw-wakeword-spike with `okay_nabu` (2 d): CPU/RAM GO/NO-GO that conditions the ENTIRE invocation path (the ROADMAP already marks it as a root dependency); includes the steady heap measurement (budget gap). (3) wakeword-training starts in parallel (audit `personal_samples` + `prepare_datasets` + first run): does not depend on the spike because the output format is identical to `okay_nabu`. (4) agent-modernization step 0: create the custom LiveKit Cloud project with API keys (blocking gap) + upgrade to `livekit-agents` 1.6.4 + `AgentServer`; without this, there is no agent-side for any protocol. PHASE B — freeze contracts (days 3-5, after the spike GO): resolve in a single document the three cross-decisions: device→agent channel = `publish_data` with retry (gate-preroll verification wins over agent-modernization), canonical tap/pre-roll sample rate (48k detector + explicit 16k vs 48k decision for the ring/stream), and gate mechanism (`StopResponse`+`TurnDetector` vs manual turn — choose one between agent-modernization and ddsd). The state machine and the SINGLE `mic_src`/`xvf_ui` refactoring (gate vs `hard_mute`) shared by both firmware areas are also specified here. PHASE C — P0 implementation (weeks 2-3): complete fw-gate-preroll-protocol (invocation task, pre-roll byte stream, incoming RPC handlers) against the already modernized agent, using tap-button as synthetic wake; in parallel, model-cost-matrix (S, 1.5-2 d) sets up the model factory on the new `agent.py` — this must go BEFORE ddsd because it fixes the model and decides if the Gemini proactivity shortcut is even evaluable; token server + explicit dispatch (decided in Phase B where it runs). At the end of the phase: integrate the trainer's es-ES model into the firmware (training + spike GO converge here) and E2E real wake. PHASE D — P1 (weeks 4-5): close aec-verification (dynamic `OP_R` switch + real `AIC3104` volume + retune) → enables barge-in → disable `half_duplex` in gate-preroll; week of Gemini Proactive Audio experiment (ddsd) as a generator of weak labels, without compromising architecture. PHASE E — complete ddsd-speaker-id (the L area, 13-15 d): goes last in the product plan because it depends on the 1.6.4 migration (done), the DoA protocol over user packets (done in C), the model fixed by the factory, the AEC verdict for double-talk, and the ~2 weeks of passive shadow mode collection that can only start when the Phase C pre-roll pipeline is in production. PHASE F — infra-selfhost (P2): at the end, after deciding registry/CI (gap), with the BVC-off vs dtln A/B testing done AFTER closing AEC (contradiction #5) and the turn detector already parameterized v1/v1-mini; meanwhile, Cloud economics are resolved by advancing IDLE (decision pending from Phase B) or with Ship $50/month as a buffer.

## Notes
- Positive convergence to highlight: fw-wakeword-spike and fw-gate-preroll-protocol arrived separately at the SAME correction to the ROADMAP (separate gate vs `hard_mute` instead of 'reuse `setMuted()`') — it is an already mature decision, it just needs to be done once and the ROADMAP updated.
- The RPC contradiction (finding #1) is the most urgent to resolve because it is a matter of fact, not opinion: simply reread `core/rpc_manager.c` from the vendored component; the gate-preroll evidence (header with comment vs absent implementation) is deeper than agent-modernization's (header only). If a future 0.3.x bump adds `rpc_invoke`, migrate the wake event to real RPC with ack.
- The LINGER cost of model-cost-matrix and the ddsd design use compatible assumptions but different justifications: the DDSD gate should be relabeled as a privacy/UX feature in the ROADMAP so that a budget cut doesn't kill it for being 'unprofitable'.
- Transversal risk unspoken by anyone: four areas (spike, gate-preroll, agent-modernization, infra) fix 'freeze C SDK version 0.3.10 and review changelog' as mitigation — there should be a single owner of dependency bumps (C SDK, esp-tflite-micro/esp-nn, livekit-agents, Speechmatics model version) because pins break in cascade.
- Cost calculations (model-cost-matrix, infra-selfhost) are a July 2026 snapshot on preview models; instrument `session_usage_updated` from the first Phase C deployment to recalculate with real data before fixing the default model.
- The aec-verification area is the only one whose uncertainty is hardware-based (does GPIO44 reach the XVF's I2S-in?) and has the cheapest test of the whole plan (10 min): do it literally first — a NO changes the verdict of barge-in, of ddsd (latency without overlap), and of the BVC A/B testing in self-host.
