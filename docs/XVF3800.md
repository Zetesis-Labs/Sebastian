# ReSpeaker XVF3800

This document describes the **XVF3800** chip on the ReSpeaker 4-mic array board and how this project controls it from the ESP32 firmware. The reference code is in [`firmware/main/xvf_dfu.zig`](../firmware/main/xvf_dfu.zig).

## Overview

The XVF3800 is a voice processing SoC from **XMOS**, integrated into the Seeed **ReSpeaker 4-mic array** board. It handles in hardware:

- **Beamforming** over the 4 microphones.
- **AEC** (acoustic echo cancellation).
- **Noise suppression** (NS).

It is controlled via **I2C at address `0x2C`**. In this project the XVF acts as an **I2S master**: it is the one that generates the I2S bus clock at **48 kHz** and delivers the already processed microphone audio to the ESP32.

## Firmware families

The XVF3800 has interchangeable firmware, and which firmware is running determines the entire behavior of the audio bus:

- **"non-master" / `i2s_dfu`** (e.g. `respeaker_xvf3800_i2s_dfu_firmware_v1.0.4`):
  the XVF is an **I2S slave**. In this project it did **not** manage to transmit the microphone audio — the mic I2S line read all zeros.

- **"inthost" / I2S-master** (`application_xvf3800_inthost-lr48-sqr-i2c-v1.0.7`):
  the XVF is an **I2S master**, clocks the bus at 48 kHz and transmits the already processed microphone audio. The suffix `lr48` indicates LR (left/right) clock at 48 kHz. **This is the firmware that the project needs.**

The goal of the project is to guarantee that the XVF always runs the "inthost" (master) family, version **1.0.7**.

## Why we do the DFU from our own firmware

Instead of flashing the XVF with an external tool or with ESPHome, the project **updates the XVF from the ESP32 firmware itself, over I2C**, using the XMOS DFU control protocol. Everything is implemented in pure Zig in `firmware/main/xvf_dfu.zig` (**without ESPHome**).

The target firmware binary (~868 KB) is **embedded** inside the ESP firmware via Zig's `@embedFile`, from `firmware/main/xvf_fw/xvf_master_1.0.7.bin`:

```zig
const firmware = @embedFile("xvf_fw/xvf_master_1.0.7.bin");
```

On boot, `xvf_dfu.ensureMaster()` does the following:

1. Adds the XVF as an I2C device (`0x2C`, 100 kHz).
2. Reads the running version (`GETVERSION`). **If it is already the target one (1.0.7), it does nothing.**
3. Otherwise: `SETALTERNATE(upgrade)`, transmits the firmware in `DNLOAD` blocks of **128 bytes**, polling `GETSTATUS` between blocks (respecting the delay that the chip returns), concludes with an empty download, sends `REBOOT`, waits, and reads the version again to verify.

**The risk is bounded**: the XVF keeps a **factory image** (`ALTERNATE_FACTORY`) that is never overwritten, so a failed upgrade **cannot leave the chip unusable (brick)**.

## The DFU protocol over I2C

The XMOS control protocol works like this:

- A command header is **written** over I2C:
  `{ resid, cmd_id, payload_len, ...payload }`.
- For **reads**, the `cmd_id` has the `READ_BIT` (`| 0x80`) set and then `{ ctrl_status, ...data }` is **read back**. A `ctrl_status == 0` indicates success.

The DFU commands live under **resid 240** (`DFU_CONTROLLER_SERVICER`):

| Command        | ID   | Direction | Use in the project |
|----------------|------|-----------|--------------------|
| `DNLOAD`       | 1    | write     | Sends a firmware block (up to 128 bytes) |
| `GETSTATUS`    | 3    | read      | Polls the DFU status between blocks |
| `SETALTERNATE` | 64   | write     | Selects the upgrade partition (`ALTERNATE_UPGRADE=1`) |
| `GETVERSION`   | 88   | read      | Reads the running firmware version |
| `REBOOT`       | 89   | write     | Reboots the XMOS into the new image |
| `READ_BIT`     | 0x80 | —         | OR-ed to the `cmd_id` for reads |

**DFU state machine** — the "ready" states that the project uses to know when it can continue sending are:

| State                     | Value |
|---------------------------|-------|
| `DFU_IDLE`                | 2     |
| `DNLOAD_IDLE`             | 5     |
| `MANIFEST_WAIT_RESET`     | 8     |

`waitReady()` polls `GETSTATUS` in a loop: the 6-byte response carries the state in `resp[5]` and a recommended delay in `resp[2..5]` (24-bit little-endian) that the firmware respects before retrying. When the state is one of the three "ready" ones, the next block can be sent.

Each `DNLOAD` block (`sendBlock`) is always transmitted as a payload of **128 bytes padded with zeros** (133 bytes of frame in total: 5 of header + 128 of data), with `req[3]` indicating the number of valid bytes. The download is concluded with an **empty block** (`sendBlock(firmware[0..0])`), after which `REBOOT` is sent and it waits ~4 s for the XVF to boot with the new firmware before reverifying the version.

## The mute mechanism

The XVF **boots MUTED**: it clocks the I2S bus but transmits **silence** until the mute is removed. This is important — the bus is alive (there is clock and there are frames) but the audio is all zeros.

The unmute is an **I2C write to the GPO-servicer on GPIO 30**:

```zig
// {resid=20 GPO servicer, cmd=1 WRITE_VALUE, len=2, gpio=30, value=0 (unmute)}
const req = [_]u8{ 20, 1, 2, 30, 0 };
```

- `resid = 20` → GPO servicer (general purpose outputs).
- `cmd = 1` → `WRITE_VALUE`.
- `len = 2` → two payload bytes (gpio and value).
- `gpio = 30` → the internal mute GPIO.
- `value = 0` → **unmute**. `value = 1` **mutes**.

The mute state can be **read back** with a read from the GPO servicer:

```zig
const req = [_]u8{ 20, 0 | READ_BIT, 6 };
// the mute bit is (resp[2] & 0x01)
```

This is done in `xvf_dfu.unmute()` during boot (which also calls `readMuteStatus()` to confirm via log that the unmute took effect).

## The two output channels

According to the XMOS datasheet, the XVF offers **two output paths** spread across the **2 I2S slots**:

- **ASR output**: taken **directly from the beamformer**, **bypassing post-processing** — therefore **without noise suppression**. It carries a configurable fixed gain intended for ASR (automatic speech recognition) engines.

- **Communications output**: **post-processed** (dereverberation + noise suppression + EQ + gain + limiter).

In the I2S of the ReSpeaker board, the mapping is:

- **LEFT slot** = processed / communications output (**with NS**).
- **RIGHT slot** = raw ASR beam (**without NS**).

### Honest caveat

- This left/right mapping comes from the **ReSpeaker wiki**, **not** from a verified primary XMOS citation.
- The XMOS datasheet **does not explicitly recommend** which channel is better for which application — literally "does not explicitly recommend which output suits which application".
- The project chose the **RIGHT/ASR beam empirically**: the NS of the LEFT channel stacked with the agent's BVC (= double noise suppression = a "tin-can" artifact). The **raw beam + a single BVC pass** sounds clean.

## Troubleshooting

**If the microphone reads all zeros**, check two things:

1. **The XVF firmware version**. If it is running the "non-master" / `i2s_dfu` family (I2S slave), it does not transmit mic audio and the I2S line reads as zeros. It must be on the "inthost" / master **1.0.7**. `ensureMaster()` should correct it on boot; check the DFU logs.
2. **The mute GPIO**. The XVF boots muted: it clocks the bus but transmits silence until the unmute (GPIO 30 = 0). Confirm with the readback of `readMuteStatus()` that the state is `UNMUTED`.
