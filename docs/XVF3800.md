# ReSpeaker XVF3800

Este documento describe el chip **XVF3800** de la placa ReSpeaker 4-mic array y
cómo este proyecto lo controla desde el firmware del ESP32. El código de
referencia está en [`firmware/main/xvf_dfu.zig`](../firmware/main/xvf_dfu.zig).

## Visión general

El XVF3800 es un SoC de procesamiento de voz de **XMOS**, integrado en la placa
Seeed **ReSpeaker 4-mic array**. Se encarga en hardware de:

- **Beamforming** sobre los 4 micrófonos.
- **AEC** (cancelación de eco acústico).
- **Supresión de ruido** (noise suppression, NS).

Se controla por **I2C en la dirección `0x2C`**. En este proyecto el XVF actúa
como **maestro I2S**: es él quien genera el reloj del bus I2S a **48 kHz** y
entrega el audio de micrófono ya procesado hacia el ESP32.

## Familias de firmware

El XVF3800 tiene firmware intercambiable, y de qué firmware esté corriendo
depende todo el comportamiento del bus de audio:

- **"non-master" / `i2s_dfu`** (p. ej. `respeaker_xvf3800_i2s_dfu_firmware_v1.0.4`):
  el XVF es **esclavo I2S**. En este proyecto **no** llegaba a transmitir el
  audio de los micrófonos — la línea I2S de mic se leía toda a ceros.

- **"inthost" / I2S-master** (`application_xvf3800_inthost-lr48-sqr-i2c-v1.0.7`):
  el XVF es **maestro I2S**, reloja el bus a 48 kHz y transmite el audio de
  micrófono ya procesado. El sufijo `lr48` indica reloj LR (left/right) a
  48 kHz. **Este es el firmware que el proyecto necesita.**

El objetivo del proyecto es garantizar que el XVF corra siempre la familia
"inthost" (maestro), versión **1.0.7**.

## Por qué hacemos el DFU desde nuestro propio firmware

En lugar de flashear el XVF con una herramienta externa o con ESPHome, el
proyecto **actualiza el XVF desde el propio firmware del ESP32, por I2C**,
usando el protocolo de control DFU de XMOS. Todo está implementado en Zig puro
en `firmware/main/xvf_dfu.zig` (**sin ESPHome**).

El binario de firmware objetivo (~868 KB) se **embebe** dentro del firmware del
ESP mediante `@embedFile` de Zig, desde
`firmware/main/xvf_fw/xvf_master_1.0.7.bin`:

```zig
const firmware = @embedFile("xvf_fw/xvf_master_1.0.7.bin");
```

En el arranque, `xvf_dfu.ensureMaster()` hace lo siguiente:

1. Añade el XVF como dispositivo I2C (`0x2C`, 100 kHz).
2. Lee la versión en ejecución (`GETVERSION`). **Si ya es la objetivo (1.0.7),
   no hace nada.**
3. En caso contrario: `SETALTERNATE(upgrade)`, transmite el firmware en bloques
   `DNLOAD` de **128 bytes**, sondeando `GETSTATUS` entre bloques (respetando el
   retardo que el chip devuelve), concluye con una descarga vacía, envía
   `REBOOT`, espera, y vuelve a leer la versión para verificar.

**El riesgo está acotado**: el XVF conserva una **imagen de fábrica**
(`ALTERNATE_FACTORY`) que nunca se sobrescribe, de modo que un upgrade fallido
**no puede dejar el chip inservible (brick)**.

## El protocolo DFU sobre I2C

El protocolo de control de XMOS funciona así:

- Se **escribe** una cabecera de comando por I2C:
  `{ resid, cmd_id, payload_len, ...payload }`.
- Para las **lecturas**, el `cmd_id` lleva puesto el bit `READ_BIT` (`| 0x80`) y
  después se **lee de vuelta** `{ ctrl_status, ...data }`. Un `ctrl_status == 0`
  indica éxito.

Los comandos DFU viven bajo el **resid 240** (`DFU_CONTROLLER_SERVICER`):

| Comando        | ID   | Dirección | Uso en el proyecto |
|----------------|------|-----------|--------------------|
| `DNLOAD`       | 1    | escritura | Envía un bloque de firmware (hasta 128 bytes) |
| `GETSTATUS`    | 3    | lectura   | Sondea el estado DFU entre bloques |
| `SETALTERNATE` | 64   | escritura | Selecciona la partición de upgrade (`ALTERNATE_UPGRADE=1`) |
| `GETVERSION`   | 88   | lectura   | Lee la versión de firmware en ejecución |
| `REBOOT`       | 89   | escritura | Reinicia el XMOS a la nueva imagen |
| `READ_BIT`     | 0x80 | —         | Se OR-ea al `cmd_id` para las lecturas |

**Máquina de estados DFU** — los estados "listos" (ready) que el proyecto usa
para saber cuándo puede seguir enviando son:

| Estado                    | Valor |
|---------------------------|-------|
| `DFU_IDLE`                | 2     |
| `DNLOAD_IDLE`             | 5     |
| `MANIFEST_WAIT_RESET`     | 8     |

`waitReady()` sondea `GETSTATUS` en bucle: la respuesta de 6 bytes lleva el
estado en `resp[5]` y un retardo recomendado en `resp[2..5]` (little-endian de
24 bits) que el firmware respeta antes de reintentar. Cuando el estado es uno de
los tres "listos", se puede enviar el siguiente bloque.

Cada bloque `DNLOAD` (`sendBlock`) se transmite siempre como un payload de
**128 bytes rellenado con ceros** (133 bytes de trama en total: 5 de cabecera +
128 de datos), con `req[3]` indicando el número de bytes válidos. La descarga se
concluye con un **bloque vacío** (`sendBlock(firmware[0..0])`), tras el cual se
envía `REBOOT` y se espera ~4 s a que el XVF arranque con el nuevo firmware
antes de reverificar la versión.

## El mecanismo de mute

El XVF **arranca silenciado (MUTED)**: reloja el bus I2S pero transmite
**silencio** hasta que se le quita el mute. Esto es importante — el bus está
vivo (hay reloj y hay tramas) pero el audio es todo ceros.

El unmute es una **escritura I2C al GPO-servicer sobre el GPIO 30**:

```zig
// {resid=20 GPO servicer, cmd=1 WRITE_VALUE, len=2, gpio=30, value=0 (unmute)}
const req = [_]u8{ 20, 1, 2, 30, 0 };
```

- `resid = 20` → servicer de GPO (salidas de propósito general).
- `cmd = 1` → `WRITE_VALUE`.
- `len = 2` → dos bytes de payload (gpio y value).
- `gpio = 30` → el GPIO interno de mute.
- `value = 0` → **unmute**. `value = 1` **silencia**.

El estado de mute se puede **leer de vuelta** con una lectura del GPO servicer:

```zig
const req = [_]u8{ 20, 0 | READ_BIT, 6 };
// el bit de mute es (resp[2] & 0x01)
```

Esto se hace en `xvf_dfu.unmute()` durante el arranque (que además llama a
`readMuteStatus()` para confirmar por log que el unmute tomó efecto).

## Los dos canales de salida

Según el datasheet de XMOS, el XVF ofrece **dos rutas de salida** repartidas en
los **2 slots I2S**:

- **Salida ASR**: tomada **directamente del beamformer**, **saltándose el
  post-procesado** — por tanto **sin supresión de ruido**. Lleva una ganancia
  fija configurable pensada para motores ASR (reconocimiento de voz).

- **Salida de comunicación**: **post-procesada** (reducción de reverberación +
  supresión de ruido + EQ + ganancia + limitador).

En el I2S de la placa ReSpeaker, el mapeo es:

- **Slot LEFT (izquierdo)** = salida procesada / de comunicación (**con NS**).
- **Slot RIGHT (derecho)** = beam ASR en crudo (**sin NS**).

### Aviso honesto (caveat)

- Este mapeo izquierda/derecha proviene de la **wiki de ReSpeaker**, **no** de
  una cita primaria verificada de XMOS.
- El datasheet de XMOS **no recomienda explícitamente** qué canal conviene para
  qué aplicación — literalmente "does not explicitly recommend which output
  suits which application".
- El proyecto eligió el **beam RIGHT/ASR de forma empírica**: la NS del canal
  LEFT se apilaba con el BVC del agente (= doble supresión de ruido = un
  artefacto "de lata"/"tin-can"). El **beam en crudo + una sola pasada de BVC**
  suena limpio.

Referencia del datasheet:
<https://www.xmos.com/documentation/XM-014888-PC/html/modules/fwk_xvf/doc/datasheet/03_audio_pipeline.html>

## Troubleshooting

**Si el micrófono se lee todo a ceros**, revisa dos cosas:

1. **La versión de firmware del XVF**. Si está corriendo la familia "non-master"
   / `i2s_dfu` (esclavo I2S), no transmite el audio de mic y la línea I2S se lee
   a ceros. Debe estar en la "inthost" / maestro **1.0.7**. `ensureMaster()`
   debería corregirlo en el arranque; revisa los logs del DFU.
2. **El GPIO de mute**. El XVF arranca silenciado: reloja el bus pero transmite
   silencio hasta el unmute (GPIO 30 = 0). Confirma con el readback de
   `readMuteStatus()` que el estado es `UNMUTED`.
