# System Architecture вҖ” Sebastian

Sebastian is a **conversational voice speaker**. The user speaks, the device captures and cleans the voice via hardware, transports it in real-time to the cloud, an agent understands it and responds with voice, and that response is played back through the speaker. The entire loop is voice-to-voice.

- **Hardware:** 4-microphone array Seeed ReSpeaker XVF3800 + XIAO ESP32-S3.
- **Backend:** LiveKit Cloud + a Python agent that uses OpenAI Realtime (speech-to-speech).
- **Firmware (application layer):** Zig on ESP-IDF v5.4 + the LiveKit C SDK (`client-sdk-esp32`).
- **Repository:** `github.com/Zetesis-Labs/Sebastian`

---

## 1. High-level data flow

```
4 microphones
    в”Ӯ
    в–ј
XVF3800 (XMOS voice DSP: beamforming + AEC + noise suppression)
    в”Ӯ  I2S (48 kHz, 32-bit, stereo)
    в–ј
ESP32-S3
    в”Ӯ  Opus / WebRTC
    в–ј
LiveKit Cloud (room)
    в”Ӯ
    в–ј
Python agent (OpenAI Realtime, speech-to-speech)
    в”Ӯ  returns to room
    в–ј
LiveKit Cloud (room)
    в”Ӯ  Opus / WebRTC
    в–ј
ESP32-S3
    в”Ӯ  I2S
    в–ј
AIC3104 Codec (DAC)
    в”Ӯ
    в–ј
Speaker
```

The key design principle is that **heavy audio cleaning occurs on the XVF3800 silicon**, not on the ESP32. The ESP32-S3 only moves PCM between I2S and the network; it does not process signal.

## 2. Hardware topology and buses

The XVF3800 is the audio brain: the 4 microphones are physically connected to it, and beamforming, AEC (acoustic echo cancellation), and noise suppression run on it. The AIC3104 is a passive codec (DAC) for the speaker output, configured via I2C only for volumes and routing (it does not generate clock).

### 2.2 I2C bus

| Signal | GPIO |
|--------|------|
| SDA    | 5    |
| SCL    | 6    |

| Device | I2C Address | Role |
|--------|-------------|------|
| XVF3800 (XMOS voice DSP) | `0x2C` | Audio brain: beamforming, AEC, noise suppression. The 4 microphones connect to it. It self-configures. |
| TLV320AIC3104 (codec) | `0x18` | Passive DAC for speaker output. Configured via I2C only for volumes/routing; it does **not** generate clock. |

The ESP32-S3 is the **I2C bus master** (`i2c_new_master_bus`, `I2C_NUM_0`, internal pull-ups enabled).

### 2.3 I2S bus (shared clock)

| Signal | GPIO | Direction | Notes |
|--------|------|-----------|-------|
| WS / LRCLK | 7  | from XVF to ESP/AIC | word-select, shared |
| BCLK       | 8  | from XVF to ESP/AIC | bit-clock, shared |
| MCLK       | 9  | вҖ” | **UNUSED** (`I2S_GPIO_UNUSED`) |
| DOUT       | 44 | ESP вҶ’ AIC3104 | audio to speaker (TX) |
| DIN        | 43 | XVF3800 вҶ’ ESP | audio from microphone (RX) |

**I2S clock Master/slave вҖ” critical point:**

- The **XVF3800** (running the "inthost" firmware) is the I2S **MASTER**: it generates the 48 kHz clock, with 32-bit slots and stereo format.
- The **ESP32-S3** is I2S **SLAVE**.
- The **AIC3104** is also a slave, on the same clock.

The bus format is standard Philips I2S (`bit_shift = true`), 48 kHz, 32-bit slots, stereo (two slots: left and right).

### 2.4 Two separate I2S peripherals

The ESP uses **TWO independent I2S peripherals**, both as slave, sharing BCLK/WS:

| Peripheral | Use | Data pin | Role |
|------------|-----|----------|------|
| `I2S_NUM_1` | RX (microphone) | DIN = GPIO43 | slave |
| `I2S_NUM_0` | TX (speaker)   | DOUT = GPIO44 | slave |

**Why two ports and not a single duplex channel:** a single shared duplex I2S channel corrupted the DMA and caused crashes. By separating microphone (RX) and speaker (TX) into two distinct peripherals, each with its consuming task, the DMA is no longer corrupted. Both remain slaves of the same BCLK/WS clock imposed by the XVF3800.

## 3. Audio pipeline вҖ” MICROPHONE (uplink)

Capture lives in `firmware/main/mic_src.zig`, a **custom `esp_capture` audio source**.

```
XVF3800 (raw ASR beam, RIGHT slot)
   в”Ӯ  I2S RX (48 kHz, 32-bit, stereo) on I2S_NUM_1
   в–ј
mic_src.zig В· read_frame()   вҶҗ reads i2s_rx DIRECTLY, paced by the consumer
   в”Ӯ   В· extracts RIGHT slot
   в”Ӯ   В· converts 32-bit вҶ’ 16-bit (right bit-shift)
   в–ј
esp_capture source  вҶ’  Opus 48 kHz mono
   в”Ӯ
   в–ј
LiveKit (publishes; resamples downstream)
```

Design characteristics:

- **Direct reading paced by the consumer:** `read_frame()` reads `i2s_rx` directly. The pipeline requests a frame and the code blocks reading exactly that amount of I2S samples. **There is no ring buffer**, so there is no drift between the XVF's clock domain and the consumer's: the XVF's 48 kHz clock and the LiveKit consumer are the same loop. This eliminates the periodic warble/"helicopter" sound produced by a free-running ring buffer.
- **Channel selection:** extracts the **RIGHT slot** (the raw ASR beam, see В§5).
- **Format conversion:** each 32-bit sample is converted to 16 bits via a right bit-shift before publishing.
- **Publishing:** the source publishes Opus at **48 kHz mono**; LiveKit resamples downstream.
- **Clock resynchronization:** on the first `read_frame()`, a `disable` + `enable` of the RX channel is performed to lock onto the already running XVF clock (the board enabled the channel before the XVF was generating a clock).

## 4. Audio pipeline вҖ” SPEAKER (downlink)

Playback is built in `firmware/main/app.zig` (`buildRenderer`).

```
Agent (audio) via LiveKit
   в”Ӯ
   в–ј
av_render  (audio_raw_fifo / audio_render_fifo, allow_drop_data)
   в”Ӯ  48 kHz, 32-bit, stereo
   в–ј
i2s render on play_dev (esp_codec_dev wrapping i2s_tx, I2S_NUM_0)
   в”Ӯ  I2S TX (DOUT = GPIO44)
   в–ј
AIC3104 (DAC)
   в”Ӯ
   в–ј
Speaker  (volume 35/100)
```

Design characteristics:

- **Chain:** `av_render` вҶ’ an i2s render on `play_dev` (an `esp_codec_dev` wrapping `i2s_tx`) вҶ’ AIC3104 вҶ’ speaker.
- **Format:** rendered at **48 kHz, 32-bit samples**, to align with the XVF's 32-bit I2S slots. Rendering at 16 bits produced misaligned and noisy output.
- **Moderate volume (35/100):** raising it too much creates acoustic feedback in the microphone.
- **Bounded latency:** `allow_drop_data` limits playback latency by dropping data instead of accumulating latency over time (`audio_render_fifo` ~0.4 s).

## 5. Channel decision: LEFT vs RIGHT

This is a deliberate design point and deserves an honest explanation.

### 5.1 What the XMOS datasheet documents

The XVF3800 outputs **two output slots** over I2S (at 48 kHz, 32 bits). According to the XMOS datasheet, there are two output paths:

- **"ASR" output** вҖ” taken directly from the beamformer. It **bypasses the post-processor**, so it does **NOT** include noise suppression. It has a configurable fixed gain, intended to feed automatic speech recognition (ASR) engines.
- **"Communication" output** вҖ” goes through full post-processing: dereverberation + noise suppression + equalization + gain + limiter. Intended for human listening.

### 5.2 The I2S slot mapping of the ReSpeaker board

On the ReSpeaker board's I2S:

- **LEFT slot** = processed / communication output (**with** noise suppression).
- **RIGHT slot** = raw ASR beam (**without** noise suppression).

> **Honest warning:** this left/right mapping comes from the **ReSpeaker wiki**, not from a verified primary XMOS citation. XMOS does **not** explicitly recommend which channel to use for which case. Treat this as empirical, not as manufacturer dogma.

### 5.3 What this project does and why

This project uses the **RIGHT slot (raw ASR beam, no NS)**, decided **empirically**:

- Using the LEFT slot (with on-chip NS) stacked that noise suppression with the agent's own BVC noise cancellation. Two NS passes in series produced a "double NS" artifact with a **tin-can** sound.
- Feeding the raw beam (right) and letting the **agent's BVC do a single NS pass** sounds clean.

In summary: **a single noise suppression** pass in the entire chain, executed by the agent. The firmware delivers the raw beam to avoid duplicating it.

> Code consistency note: some comment in `app.zig`/`buildCapturer` mentions the LEFT slot; the actual implementation in `mic_src.zig` takes the **RIGHT** slot (`read_buf[i*2 + 1]`), which is the current and correct behavior according to this decision.

## 6. Agent side (Python)

File: `agent/agent.py`, on top of `livekit-agents`.

- **`Sebastian(Agent)` class** with instructions in **Spanish** (brief and natural voice assistant, no markdown or lists because it is only heard).
- **Model:** OpenAI Realtime (`openai.realtime.RealtimeModel`), `"alloy"` voice, speech-to-speech.
- **Noise cancellation:** `noise_cancellation.BVC()` in the `RoomInputOptions` вҖ” this is the **only** NS pass in the chain (see В§5).
- **Audio verification:** records the incoming microphone track to `/tmp/sebastian_rx.wav` (16 kHz mono) to be able to listen to what the agent actually receives. It only records audio tracks whose `participant.identity` contains `esp32`.
- **Startup:** connects to the room, and `generate_reply` greets and introduces itself in one sentence.

Operational constraints:

- It must run **a single agent process**. Duplicate processes cause it to "talk to itself".
- The agent **auto-dispatches only to a fresh room**.

Execution modes:

```bash
uv run agent.py dev      # local development, hot-reload
uv run agent.py start    # production worker
```

## 7. Firmware boot sequence

`app_main()` in `firmware/main/app.zig`:

1. `livekit_system_init()`.
2. `board.init()` вҖ” I2C, I2C scan, AIC3104 configuration, the two I2S channels (TX/RX slave), and the playback and capture `esp_codec_dev` instances.
3. `xvf_dfu.ensureMaster()` вҖ” puts the XVF into the firmware that makes it the **I2S master**.
4. `xvf_dfu.unmute()` вҖ” the XVF boots muted; it is cleared so the microphone transmits.
5. Registration of audio codecs (default decoder/encoder).
6. `buildCapturer()` вҖ” creates the custom microphone source (`mic_src`).
7. `buildRenderer()` вҖ” creates the playback pipeline (`av_render` вҶ’ I2S TX вҶ’ AIC3104).
8. WiFi (no modem-sleep, `WIFI_PS_NONE`, for real-time audio) вҶ’ SNTP (valid clock before TLS) вҶ’ `joinRoom()`.
9. `joinRoom()` вҖ” publishes Opus 48 kHz mono audio from the capturer and subscribes to room audio towards the renderer.

## 8. Summarized design decisions

| Decision | Reason |
|----------|--------|
| All heavy cleaning on the XVF3800 | The ESP32 only moves PCM; it does not process signal. |
| XVF3800 as I2S master; ESP as slave | The XVF's "inthost" firmware imposes the 48 kHz / 32-bit clock. |
| Two I2S peripherals (RX on `NUM_1`, TX on `NUM_0`) | A single duplex channel corrupted the DMA and caused crashes. |
| Consumer-paced capture, no ring buffer | Avoids clock domain drift (warble/"helicopter" sound). |
| Render at 32 bits | Aligns with the XVF's 32-bit slots; 16 bits resulted in noisy output. |
| Moderate speaker volume (35/100) | Avoids acoustic feedback in the microphone. |
| RIGHT slot (raw ASR beam) + agent BVC | A single NS pass; avoids the "tin-can" of double NS. |
| Single agent process, fresh room | Duplicates cause it to "talk to itself". |
